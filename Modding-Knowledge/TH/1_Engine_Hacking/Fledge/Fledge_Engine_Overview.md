# The Surge (FLEDGE Engine) - การวิจัยระบบแปลภาษาไทย และ บทสรุปหลังการพัฒนา (Post-Mortem)

เอกสารนี้รวบรวมความพยายามในการทำวิศวกรรมย้อนกลับ (Reverse-engineering) เพื่อสร้างม็อดภาษาไทยสำหรับเกม **The Surge** ซึ่งทำงานบนเอนจินพิเศษที่ชื่อว่า **FLEDGE Engine**

แม้ว่าโปรเจกต์นี้จะถูกระงับไปก่อนที่จะสามารถแสดงผลข้อความภาษาไทยในเกมได้อย่างสมบูรณ์แบบ แต่การวิจัยครั้งนี้ได้ค้นพบข้อมูลเชิงลึกมากมายเกี่ยวกับวิธีที่เอนจิน FLEDGE จัดการการแพ็กข้อมูล, การแสดงผลข้อความ, และการจัดการหน่วยความจำ เราจัดทำเอกสารนี้ขึ้นด้วยความหวังว่าม็อดเดอร์รุ่นหลังจะสามารถนำไปพัฒนาต่อยอดจากจุดที่เราหยุดไว้ได้

## 1. โครงสร้างข้อมูล และ การจัดแพ็กเกจ (Data Structure & Packaging)

### ระบบไฟล์ `.dat` และ `.toc`
- ไฟล์ทรัพยากรของเกมจะถูกเก็บไว้ในไฟล์ `.dat` (เช่น `data_bin_0.dat`)
- ไฟล์ `data_bin.toc` ทำหน้าที่เป็นสารบัญ (Table of Contents) ซึ่งประกอบไปด้วยค่าแฮช (hashes), ตำแหน่งออฟเซ็ต (offsets), และขนาดไฟล์ที่ถูกบีบอัด
- **ข้อจำกัดสำคัญ:** ไฟล์ TOC มีการตรวจสอบความถูกต้องอย่างเข้มงวด หากมีการแก้ไขไฟล์ `data_bin.toc` แม้แต่นิดเดียว เกมจะเด้งหลุด (Crash) ทันทีตอนเริ่มเกม ซึ่งหมายความว่าเราไม่สามารถเพิ่มไฟล์ใหม่, เปลี่ยนออฟเซ็ต, หรือเปลี่ยนขนาดข้อมูลที่ถูกบีบอัดแล้วได้เลย

### การบีบอัดแบบ LZ4
- ทรัพยากรที่อยู่ข้างในไฟล์ `.dat` จะถูกบีบอัดแยกกันด้วย **รูปแบบ LZ4 Block** (โดยเฉพาะ `lz4.frame` ร่วมกับ `BLOCKSIZE_MAX256KB`)
- เนื่องจากมีระบบป้องกันในไฟล์ TOC วิธีเดียวที่ใช้ในการม็อดได้คือ **การเขียนทับไฟล์ในตำแหน่งเดิม (in-place replacement)** 
- คุณจะต้องนำไฟล์ที่ดัดแปลงมาบีบอัดใหม่ และถ้ามันมีขนาดเล็กกว่าไฟล์ต้นฉบับ คุณจะต้องทำการถมที่ว่างที่เหลือ (Padding)
- **วิธีการถมที่ว่าง (Padding):** การถมที่ด้วยไบต์ศูนย์ (`0x00`) ทำงานได้ดีมากกับไฟล์ส่วนใหญ่ (เช่น `fonts.bin` และ `english.bin`) อย่างไรก็ตาม การพยายามถมที่ด้วยระบบ LZ4 Skippable Frames กลับทำให้เอนจินเกมค้างหรือหลุดเมื่อเจอไฟล์บางตัว เช่น `russian.bin`

## 2. การจัดการข้อความและสตริง (Text & String Management)

### ไฟล์ข้อความ (`.bin`)
- ข้อความและคำแปลในเกมจะถูกเก็บอยู่ในไฟล์ไบนารี เช่น `english.bin`, `russian.bin` เป็นต้น
- โครงสร้างของมันค่อนข้างเรียบง่าย: มี Header แบบ 32-bit integer (โดยที่บิตซ้ายสุด `0x80000000` ระบุว่าเป็นสตริงข้อความ และ 31 บิตที่เหลือคือความยาวของข้อความ) ตามด้วยข้อความแบบ UTF-8 และทำการถมไบต์ให้ลงตัวด้วย 4-byte alignment
- **การแก้ไขข้อความ:** เราเขียนสคริปต์ Python สำเร็จแล้ว (`rebuild_v15.py` และ `deploy_english_cyrillic.py`) ที่ช่วยในการแตกไฟล์ `english.bin` ออกมา, แทรกข้อความภาษาไทย UTF-8 ลงไป (โดยอาจจะต้องตัดคำหรือเว้นวรรคให้ขนาดไฟล์หลังบีบอัดเป๊ะเท่าเดิม), จากนั้นทำการบีบอัดกลับ และถมไบต์ด้วยศูนย์

### แนวทาง "การเข้ารหัสแบบ Cipher"
เนื่องจากการไปแก้รหัส ID ใน `fonts.bin` จะทำให้ระบบดัชนีภายในพัง วิธีที่ดีที่สุดที่เราคิดได้คือแนวทาง "Cipher String Patching" (แปลงข้อความเป็นรหัสลับ):
1. แปลข้อความจากอังกฤษเป็นไทย
2. แปลงรหัสอักขระไทย (0x0E01+) ไปสวมรอยอยู่ในกลุ่มภาษาที่เกมรองรับ (เช่น ภาษารัสเซีย/Cyrillic `0x0400+` หรือ ญี่ปุ่น `0x3041+`)
3. นำข้อความที่ถูกสวมรอยแล้ว ฉีดกลับเข้าไปในไฟล์ `english.bin` หรือ `russian.bin`
4. วาดรูปร่างฟอนต์ภาษาไทย ไปทับในตำแหน่งของตัวอักษรรัสเซียหรือญี่ปุ่น บนรูปภาพฟอนต์ (Font atlas) ของเกม

## 3. ระบบแสดงผลฟอนต์ (SDF)

### `fonts.bin`
- เป็นไฟล์ที่เก็บเมทริกซ์และพิกัด UV ของตัวอักษรแต่ละตัว
- ตัวอักษรแต่ละตัวจะถูกนิยามด้วยข้อมูลขนาด 32 ไบต์: `[4-byte Codepoint] [7x 4-byte Floats สำหรับระบุแกน X, Y, W, H, X-Offset, Y-Offset, X-Advance]`
- **ปัญหาการเรียงลำดับดัชนี (The Indexing Problem):** การไปแก้รหัส Codepoint ดื้อๆ ในชุดข้อมูล 32 ไบต์ จะทำให้ฟอนต์แสดงผลเป็นช่องว่าง นั่นแปลว่าเอนจินของเกมนี้น่าจะใช้โครงสร้างดัชนีแบบอื่นก่อนหน้านี้ (เช่น hash map หรือ binary search tree) เพื่อค้นหาตัวอักษร การไปแก้รหัสในตารางสุดท้ายโดยตรงจึงทำให้เกมหาฟอนต์ไม่เจอ

### รูปภาพฟอนต์ / Font Atlases (DDS)
- เอนจินใช้ระบบการเรนเดอร์แบบ **Signed Distance Field (SDF)** เพื่อให้ข้อความคมชัดในทุกความละเอียดหน้าจอ
- ภาพฟอนต์ถูกเก็บในรูปแบบไฟล์ DDS ที่ถูกบีบอัดด้วย LZ4 (เช่น `arial_sdf`, `fira_sans_regular_sdf`, `nanum_gothic_sdf`)
- ภาพฟอนต์มักจะมีขนาด 2048x1024 หรือ 2048x2048 ในรูปแบบ `DXT5` หรือ `R8/A8`
- การจะเปลี่ยนรูปร่างฟอนต์ได้นั้น คุณจะต้องสร้างภาพ SDF ของฟอนต์ไทย และนำไปตัดแปะทับลงในรูปภาพฟอนต์ DDS ของเกม (ในช่องของตัวอักษรภาษารัสเซียหรือญี่ปุ่น)

## 4. ข้อผิดพลาดและอุปสรรคที่พบ (Failures & Roadblocks)

1. **การแพตช์หน่วยความจำผ่าน DLL:** เราพยายาม hook เข้าไปใน `dinput8.dll` และสแกนหน่วยความจำของตัวเกมเพื่อเข้าไปสลับข้อความ UTF-16 สดๆ ในแรม แม้ว่า DLL จะสามารถหาข้อความและสลับค่าในแรมได้สำเร็จ แต่ข้อความ UI ในเกมกลับยังแสดงผลเป็นภาษาอังกฤษเหมือนเดิม บ่งบอกว่าเอนจิน FLEDGE อาจจะทำการประมวลผลข้อความแปลงเป็นภาพไปแล้วตั้งแต่ต้นๆ (A8 UI surfaces) หรือมีบัฟเฟอร์เก็บข้อความลับอีกชั้นที่เราหาไม่เจอ
2. **ตัวอักษรหายเมื่อใช้ Cipher:** เมื่อเราแทรกข้อความด้วยอักษรรัสเซียหรือญี่ปุ่นเข้าไปใน `english.bin` ตัวเกมกลับแสดงข้อความเป็นความว่างเปล่า นั่นแปลว่าเวลาที่เราตั้งภาษาของเกมเป็นภาษาอังกฤษ เกมจะโหลดรูปร่างฟอนต์เฉพาะภาษาที่จำเป็นเข้าการ์ดจอ (VRAM) และเตะฟอนต์รัสเซียหรือญี่ปุ่นทิ้งไปเพื่อประหยัดแรม
3. **เกมแครชเมื่อใช้ภาษารัสเซีย:** เราพยายามฉีดข้อความเข้า `russian.bin` แล้วแก้ไฟล์ `language.ini` บังคับให้เกมเปิดด้วยภาษารัสเซีย แต่เกมแครช สาเหตุน่าจะมาจากระบบตรวจจับค่า Hash และขนาดไฟล์ที่ทำงานเข้มงวดมากสำหรับไฟล์นี้ หรืออาจจะเป็นเพราะมันไม่ยอมรับการถมที่ (Zero-padding) ในรูปแบบ LZ4 ของไฟล์รัสเซีย

## 5. ข้อเสนอแนะสำหรับม็อดเดอร์ในอนาคต

หากใครต้องการสานต่อโปรเจกต์นี้ นี่คือเส้นทางที่แนะนำครับ:
1. **ทำความเข้าใจระบบดัชนีของ `fonts.bin`:** ถอดรหัสโครงสร้าง Header ของ `fonts.bin` ให้ได้ เพื่อทำความเข้าใจว่าเอนจินมันโยง Codepoint เข้ากับชุดข้อมูล 32 ไบต์ยังไง ถ้าคุณสามารถอัปเดตดัชนีนี้ได้ คุณก็สามารถเพิ่มรหัสภาษาไทย (`0x0E01+`) ลงไปตรงๆ ได้เลย
2. **การฉีดรูปภาพ DDS:** สร้างระบบที่เสถียรในการแตกไฟล์บีบอัด LZ4 DDS, เอาภาพฟอนต์ SDF ไทยไปแปะทับตัวอักษรภาษาอังกฤษแบบ Extended (พวกตัวมีขีดๆ ด้านบน ซึ่งตัวพวกนี้ *จะถูกโหลด* แน่นอนในภาษาอังกฤษ) และทำการบีบอัดกลับให้ขนาดเป๊ะเท่าเดิม
3. **ทำ Cipher ทับภาษาอังกฤษ:** แทนที่จะสวมรอยไปอยู่ภาษารัสเซียหรือญี่ปุ่น (ซึ่งเกมมันไม่โหลด) ให้เอาภาษาไทยไปสวมรอยอยู่ในตัวอักษรยุโรป Extended Latin แทน (เช่น `À, Á, Â, Ã, Ä, Å`) เพื่อการันตีว่ามันจะมีภาพฟอนต์ขึ้นมาแน่ๆ ในโหลดภาษาอังกฤษ

## สคริปต์ที่มีให้ในโปรเจกต์นี้
- `patch_safe.py`: แก้ไขไฟล์ `fonts.bin` เพื่อโยงค่า ID ได้อย่างปลอดภัย
- `deploy_english_cyrillic.py` & `deploy_english_japanese.py`: สคริปต์สวมรอย Cipher ที่จัดการเรื่องขนาด LZ4 ให้เป๊ะเท่าเดิมและการถมไบต์ด้วยศูนย์
- `dinput8_thai.cpp`: โค้ดสำหรับทำ DLL แพตช์สลับข้อความในแรมแบบสดๆ (ที่ลองแล้วล้มเหลว)
